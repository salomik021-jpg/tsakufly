# TsakuFly — რეალური დეველოპერული Deploy (GitHub + Netlify)

ეს არის ის მიდგომა, რომელსაც რეალურად იყენებენ დეველოპერები: კოდი ინახება **GitHub**-ზე (ვერსიების კონტროლით — ყოველთვის გაქვთ ისტორია და შესაძლებელია უკან დაბრუნება), ხოლო **Netlify** ავტომატურად აქვეყნებს საიტს ყოველ ცვლილებაზე. აღარასდროს მოგიწევთ ხელით ფაილის თავიდან ატვირთვა — უბრალოდ შეინახავთ ცვლილებას GitHub-ზე, და Netlify თავად აახლებს ცოცხალ საიტს ~30 წამში.

კოდის წერა არ დაგჭირდებათ — მხოლოდ ღილაკებზე დაჭერა ბრაუზერში.

**რა დაგჭირდებათ:** `index.html` ფაილი (ჩატში მიმაგრებული, უკვე მზადაა) და ორი უფასო ანგარიში — GitHub და Netlify.

---

## ნაწილი A — GitHub რეპოზიტორიის შექმნა

### ნაბიჯი 1 — გახსენით ანგარიში GitHub-ზე

გადადით **github.com** → **Sign up** → შეავსეთ ელფოსტა, პაროლი, username. უფასოა.

### ნაბიჯი 2 — შექმენით ახალი რეპოზიტორია (თქვენი პროექტის "სახლი")

1. ზედა მარჯვენა კუთხეში დააჭირეთ **+** ნიშანს → **New repository**
2. **Repository name**: ჩაწერეთ `tsakufly`
3. აირჩიეთ **Public**
4. დანარჩენი ველები დატოვეთ ისე, როგორც არის
5. დააჭირეთ **Create repository**

### ნაბიჯი 3 — ატვირთეთ თქვენი ფაილი

ახლადშექმნილ, ცარიელ რეპოზიტორიაზე:

1. დააჭირეთ **uploading an existing file** ბმულს (ან **Add file → Upload files**)
2. გადაათრიეთ `index.html` ფაილი ამ ფანჯარაში
3. ქვემოთ, **Commit changes** ველში დატოვეთ ნაგულისხმევი წარწერა (ან დაწერეთ, მაგ. „პირველი ვერსია")
4. დააჭირეთ მწვანე ღილაკს **Commit changes**

თქვენი კოდი ახლა GitHub-ზეა შენახული.

---

## ნაწილი B — Netlify-ის დაკავშირება GitHub-თან

### ნაბიჯი 4 — გახსენით ანგარიში Netlify-ზე

გადადით **app.netlify.com** → **Sign up** (თუ Drop-ის მეთოდი უკვე გამოცადეთ და ანგარიში გაქვთ, უბრალოდ შედით).

### ნაბიჯი 5 — დაამატეთ ახალი პროექტი

1. დაშბორდზე დააჭირეთ **Add new project**
2. აირჩიეთ **Import an existing project**
3. აირჩიეთ **GitHub** (არა GitLab/Bitbucket)
4. დაადასტურეთ წვდომა (**Authorize Netlify**) — გადაგამისამართებთ GitHub-ზე ავტორიზაციისთვის

### ნაბიჯი 6 — აირჩიეთ თქვენი რეპოზიტორია

ჩამონათვალში (ან საძიებო ველში) იპოვეთ **tsakufly** და დააჭირეთ მას.

### ნაბიჯი 7 — Deploy-ის პარამეტრები

გამოჩნდება პატარა ფორმა — ჩვენს შემთხვევაში (უბრალო HTML ფაილი, build პროცესი არ სჭირდება):

| ველი | რა ჩაწეროთ |
|---|---|
| Branch to deploy | `main` (ნაგულისხმევი) |
| Build command | **ცარიელი** დატოვეთ |
| Publish directory | **`.`** (წერტილი — ანუ repo-ს ძირი) |

დააჭირეთ **Deploy site**.

### ნაბიჯი 8 — დაელოდეთ

~20–40 წამში დეპლოის სტატუსი გახდება მწვანე (✓ Published). დააჭირეთ ბმულს — ეს არის თქვენი ცოცხალი საიტი.

---

## ნაწილი C — მისამართის მორგება

**Site settings → Change site name** → ჩაწერეთ სასურველი სახელი, მაგ. `tsakufly` → მისამართი გახდება:

`https://tsakufly.netlify.app`

---

## ნაწილი D — როგორ განაახლოთ საიტი მომავალში

აქ არის ამ მიდგომის მთავარი უპირატესობა Drop-ის მეთოდთან შედარებით:

1. GitHub-ზე, თქვენს `tsakufly` რეპოზიტორიაში, გახსენით `index.html`
2. დააჭირეთ ფანქრის ხატულას (**Edit this file**)
3. შეცვალეთ, რაც გჭირდებათ
4. ჩამოსქროლეთ ქვემოთ და დააჭირეთ **Commit changes**
5. მოცადეთ ~30 წამი — Netlify ავტომატურად ამოიცნობს ცვლილებას და თავად გამოაქვეყნებს ახალ ვერსიას ცოცხალ საიტზე

არაფრის ხელახლა ატვირთვა აღარ გჭირდებათ — ეს არის სწორედ ის „რეალური workflow", როცა ცვლილება ერთხელ ინახება ერთგან (GitHub), და ავტომატურად ვრცელდება ყველგან.

---

## ნაწილი E (მომავალში) — საკუთარი დომენი

როცა მოინდომებთ, მაგ. `tsakufly.ge`:

1. შეიძინეთ დომენი რეგისტრატორთან (Namecheap, GoDaddy, ან `.ge`-სთვის ქართული რეგისტრატორი)
2. Netlify-ში: **Domain settings → Add a domain**
3. Netlify გაჩვენებთ, რა DNS ჩანაწერები დაამატოთ დომენის რეგისტრატორთან — ჩვეულებრივ 10-15 წუთში აქტიურდება

---

## (სურვილისამებრ) კომპიუტერზე მუშაობა Git-ით

ზემოთ აღწერილი ყველაფერი ბრაუზერშივე კეთდება. თუ მომავალში მოისურვებთ ცვლილებების კომპიუტერიდან, ტექსტურ რედაქტორში (მაგ. VS Code) გაკეთებას, დაგჭირდებათ:

- **Git**-ის დაყენება (git-scm.com) და/ან **GitHub Desktop** (desktop.github.com) — გრაფიკული აპლიკაცია, რომელიც command line-ს გვერდს უვლის
- საბაზისო ნაკადი: `Clone` რეპოზიტორია კომპიუტერზე → შეცვალეთ ფაილები → **GitHub Desktop**-ში დააჭირეთ **Commit** → **Push origin**

ეს ეტაპი სავალდებულო არაა — ბრაუზერით მუშაობა (ნაწილი D) სავსებით საკმარისია.

---

### ხშირი კითხვები

**რატომ Publish directory არის `.` და არა `dist` ან `build`?**
რადგან ეს არის უბრალო, ერთი HTML ფაილი — არანაირი "აწყობის" (build) პროცესი არ სჭირდება. `dist`/`build` საჭიროა მხოლოდ მაშინ, როცა პროექტი React/Vue-ს მსგავს framework-ს იყენებს.

**შემიძლია იგივე GitHub რეპოზიტორია გამოვიყენო Close Command Center-ისთვის ან Aura-სთვისაც?**
დიახ, მაგრამ სჯობს თითოეულისთვის ცალკე რეპოზიტორია და ცალკე Netlify საიტი შექმნათ — ასე დამოუკიდებლად განახლდება თითოეული.

**რა განსხვავებაა წინა (Drop) მეთოდსა და ამას შორის?**
Drop მეთოდი სწრაფია, მაგრამ ყოველი ცვლილებისთვის ხელახლა ფაილის გადათრევა გჭირდებათ. ეს მეთოდი ინახავს ისტორიას GitHub-ზე და ავტომატურად აქვეყნებს ცვლილებებს — სწორედ ეს არის „რეალური" დეველოპერული პროცესი.
